proto.dtd

This DTD defines the "proto" dialect for the PixelZoo XML game format, wherein all particle types and variables (vars) are referred to by name.

This dialect can be compiled to the "game" dialect, using the zoocompiler.pl program.

<goal> Child of not, pass, repeat, target, game, area, rule, cage, next, trigger, fail, or, and, cached, p, default

PixelZoo Goal is a domain-specific mini-language for testing challenge conditions and creating side effects like rewards, hints, game events, and so forth. A Goal node in the XML tree represents a single expression node in this mini-language.

A Goal expression is a function, with side effects, that returns a Boolean (true or false) value.

Each Goal expression operates on a given area of the Board (i.e. a subset of the Board's cells). By default, this area is inherited from the parent Goal expression, and the default area for the root Goal is the entire Board.

<exit> Child of game

The exit portal for the Board, enabling Lemmings-style play. The exit portal is initially in the "PortalWaiting" state. When placed in the "PortalCounting" state (as a side-effect of a Goal expression), the portal counts the number of times that particles of a given type ("type") are written to it. (The actual write operations are intercepted, so the particles appear to drain from the Board.)

The Board has a clock that is measured in microticks ("t"), a grammar of update rules ("grammar"), an optional move queue ("queue") and random number seed ("seed").

Cells are empty (zero) by default, unless specifically initialized ("init").

<particle> Child of grammar

An elementary particle in the game, identified by a unique 16-bit type ID ("type"). Each particle has its own update rate ("rate") and rule ("rule"), formulae for determining the HSB color values ("hue", "sat" and "bri"), and optionally a message dispatch table ("dispatch").

Any given instance of a particle has 48 additional bits of instance-specific state. These 48 bits are conventionally split amongst a series of Vars: named bit-fields of different widths ("vars").

By default, particles are updated asynchronously and randomly, but they optionally may be synchronized ("sync").

<rule> Child of pass, dispatch, fail, subrule, particle, default, p, next

A particle update rule. There are six types (of which only four are public), plus four constructs that are provided by the compiler for convenience, constituting an extremely minimal and predictable programming language for cellular automata.

The four compiler constructs are bind, nop, huff, and ball:

  • bind temporarily associates a name in the compiler's namespace with a neighboring cell location and performs a switch rule on the particle type at that location.
  • nop does nothing (implemented by the compiler using a dummy modify rule).
  • huff allows for a probability distribution over an arbitrary number of rules (implemented by the compiler using random rules and a Huffmann tree).
  • ball is a shortcut for a goal rule that places a text balloon.

The four public rules are switch, modify, random, and deliver:

  • switch (read a Var from any named local cell, look up corresponding rule in a table, execute that rule).
  • modify, effectively a load-add-store instruction (read a Var from any local cell, add or subtract a certain amount, store value in Var of any local cell).
  • random, introducing stochasticity (executes one of two child rules by flipping a biased coin).
  • deliver, which sends a 64-bit message to a local cell (passing control to an update rule in that particle's dispatch table).

The two private (internally-used) rules are goto and goal:

  • goto, used not to loop, but simply to save space, (passes control to a named subroutine, which never returns),
  • goal, allowing a callback hook for in-game feedback and effects (tests a Goal, if the Board has an associated Game).

<modify> Child of pass, fail, target, default, p, rule, next

Modify rule.

Writes a value to a destination cell ("dest"). The precise behavior depends on the child tags; there are two possible options:

  1. A Var is read from a source cell ("src"), a specified offset ("inc") is added, and the resulting value is written to a specific Var of the destination cell.
  2. The type and all Vars of the destination cell are set to the specified values ("set").

The "dest" and (if specified) "src" cells are referred to by name, and must have first been bound by an outer bind rule.

After writing to the destination cells, the (optional) "next" rule is executed.
